Technical Resources Document: Codex CLI Agent Configuration for AutoResearch-RL Pipeline1. Architectural Context and Pipeline OverviewThe convergence of large language models and reinforcement learning has precipitated a paradigm shift in how neural architecture search and hyperparameter optimization are conducted. The AutoResearch-RL framework epitomizes this shift, deploying a Proximal Policy Optimization (PPO) meta-agent to autonomously propose, evaluate, and commit structural code modifications to a target machine learning training script. Designed specifically to dominate the rigorous constraints of the Parameter-Golf challenge, the system operates as a perpetual, continuous Markov Decision Process (MDP) where the editable state is strictly confined to a single file, typically train_gpt.py. Within this framework, the autonomous agent is bound by severe environmental restrictions: a 16-megabyte maximum compressed artifact size encompassing both source code and quantized weights, a strict ten-minute training temporal budget on an 8xH100 GPU cluster, and an evaluation metric optimizing exclusively for the lowest validation bits-per-byte (val-bpb) on a fixed FineWeb dataset slice.To orchestrate this perpetual research loop without human intervention, the OpenAI Codex Command Line Interface (CLI) serves as the central intelligent harness. Codex operates as a cross-platform software agent capable of traversing directories, reading abstract syntax trees, issuing shell commands, invoking external network tools, and utilizing the Model Context Protocol (MCP) to manage the complex state of long-horizon experiments. By layering a robust terminal user interface (TUI), a headless execution engine invoked via codex exec, and comprehensive support for both local Open Source Software (OSS) models and cloud-based reasoning engines such as gpt-5.3-codex, the CLI provides the necessary infrastructure to automate the AutoResearch-RL pipeline.This technical resources document synthesizes the operational configurations, durable guidance protocols, sandbox security paradigms, prompting structures, and execution wrappers required to deeply integrate the Codex CLI into the AutoResearch-RL infrastructure. The subsequent sections provide a comprehensive guide to deploying this autonomous research agent, ensuring that the integration maximizes experimental throughput while maintaining strict adherence to the competition's security and computational boundaries.2. The Parameter-Golf Challenge DirectivesThe objective function of the AutoResearch-RL pipeline is intrinsically tied to mastering the Parameter-Golf challenge. The parameters of this specific environment dictate exactly what the Codex agent must optimize, establish the boundaries of its search space, and define the metrics by which its iterations are judged.The challenge evaluates models based on validation bits-per-byte (val-bpb), which is calculated on a 50,000-document slice of the fixed FineWeb validation dataset. As established by Shannon's information theory, data compression and probabilistic prediction are mathematically equivalent; the cross-entropy loss of a language model is directly proportional to its compression efficiency. Measuring bits-per-byte normalizes the evaluation score across different tokenizers, ensuring that researchers cannot achieve artificial performance gains merely by inflating vocabulary sizes. The current naive baseline score stands at 1.2244 BPB, utilizing a baseline architecture of 9 layers and a 512-dimension hidden state. Submissions must demonstrate a statistically significant improvement of at least 0.005 nats (typically averaged over three separate random seeds) to establish a new state-of-the-art record.The most rigorous constraint imposed on the Codex agent is the 16-megabyte capacity limit, which is strictly defined as 16,000,000 decimal bytes, rather than 16 MiB (16,777,216 bytes). This cap is calculated by summing the raw UTF-8 bytes of the train_gpt.py script and the zlib-compressed bytes of the INT8-quantized weights generated during training. To guide the agent around this severe limitation, prompt instructions must continually mandate weight quantization and selective parameter pruning.The environment also enforces a strict 10-minute training limit and an additional 10-minute evaluation limit on an 8xH100 SXM topology. Because wall-clock time fluctuates due to thermal throttling across GPU generations, the agent must be instructed to optimize the data-loading pipeline and implement compute-efficient techniques.Optimization TechniqueEstimated BPB ImprovementComplexity LevelImplications for Codex Agent PlanningCAGE QAT Gradient0.002 - 0.005Low-moderateAgent must implement Quantization-Aware Training within train_gpt.py without bloating code size.Iteration-Free NS (IFNSO)0.002 - 0.005Very lowEasy implementation for Codex, providing immediate reward signals during early RL iterations.V:N:M Activation Sparsity0.005 - 0.015Moderate-highRequires the agent to generate custom attention kernels or mask matrices, demanding high reasoning effort.Sliding Window Evaluation0.032LowAgent can improve compression purely during the evaluation phase by providing richer contextual windows.During evaluation, the competition strictly prohibits external downloads, network calls, or access to training datasets. The resulting artifact must be fully self-contained. Therefore, the Codex CLI must be configured to enforce absolute network isolation during its command execution phases, ensuring that any code generated by the agent attempting to call external libraries or fetch additional data fails immediately, thereby preventing the submission of an invalid model.4. Comprehensive Codex CLI Configuration (config.toml)The Codex CLI relies on a highly granular, layered configuration system powered by TOML. Configuration resolution follows a strict precedence order: CLI command-line flags and overrides supersede named profile values; profiles override project-scoped .codex/config.toml files; project files override the global user configuration at ~/.codex/config.toml; and user configurations override system-wide defaults located at /etc/codex/config.toml on Unix systems. For security reasons, Codex will only load project-scoped configuration files if the directory is explicitly marked as trusted; otherwise, it falls back to the user configuration.For the AutoResearch-RL environment, the configuration must enforce the highest possible reasoning effort to handle complex mathematical code modifications, permit workspace modifications to allow the AST diff parser to mutate files, define the telemetry storage directories, and establish the default Large Language Model parameters.4.1 Configuration SchemaThe following represents the canonical .codex/config.toml configuration designed specifically to be placed at the root of the AutoResearch-RL repository. Security, Sandboxing, and OS-Level PoliciesBecause the AutoResearch-RL agent executes code autonomously, mutates files dynamically, and generates complex logic independent of human oversight, securing the pipeline is paramount. Unbounded agents pose a severe risk of executing destructive commands, deleting critical repository history, or exfiltrating sensitive environment variables.The Codex CLI does not rely solely on prompt-based guardrails; it enforces strict, OS-level constraints to quarantine the agent's operations. On macOS environments, it utilizes Seatbelt policies executed through sandbox-exec with specific profiles that correspond to the chosen sandbox mode. On Linux and distributed GPU clusters—where AutoResearch-RL is typically deployed—it leverages a bubblewrap pipeline combined with seccomp (secure computing mode) to aggressively restrict system calls and filesystem bindings.When the sandbox_mode is set to workspace-write, the agent is granted permission to mutate files only within the current working directory, while paths outside this root remain strictly read-only. Even within the writable root, the sandbox inherently protects version control systems; directories such as .git, .codex, and .agents are enforced as recursive read-only paths, preventing the agent from corrupting the repository's history or maliciously altering its own operational configuration.The approval_policy layer functions as a secondary safety net. While setting the policy to never is required for unsupervised overnight training, operators can utilize the [approval_policy.granular] table in the config.toml to maintain a baseline level of safety. For example, setting sandbox_approval = false guarantees that even in an automated run, the agent cannot unilaterally decide to escalate its privileges beyond the workspace.6. Durable Guidance Specifications (AGENTS.md)The Codex CLI dynamically constructs its system prompt through an instruction chain mechanism before initiating any task. This chain relies on durable guidance files, primarily an AGENTS.md file, which encodes project-specific engineering conventions, setup commands, and strict negative boundaries.Research indicates that the quality and origin of these guidance files significantly impact agent performance. An ETH Zurich study demonstrated that utilizing Large Language Models to auto-generate context files actually reduced task success rates by approximately 3%, increased inference costs by over 20%, and necessitated additional reasoning steps to correct cascading errors. Conversely, human-curated context files provided a measurable performance gain. Therefore, the AGENTS.md file for AutoResearch-RL must be meticulously crafted by the lead developer, explicitly structuring rules for machine parsing rather than human readability.Codex discovers guidance files by walking down from the project root to the current working directory, concatenating files separated by blank lines. Files positioned deeper in the directory structure override earlier guidance, allowing for fine-grained, localized instructions. For the AutoResearch-RL framework, however, a monolithic AGENTS.md at the repository root is optimal, given the specific, cross-cutting nature of the val-bpb objective and the global 16-megabyte constraint.6.1 AutoResearch-RL AGENTS.md ImplementationThe following AGENTS.md scaffold provides the exact durable guidance required by the Codex CLI to operate the AutoResearch-RL pipeline effectively. Codex Prompt Engineering Best PracticesWhile the durable guidance files establish the baseline context, the specific prompts fed into the Codex CLI dictate the tactical execution of the research loop. The OpenAI prompting guide dictates that a highly effective prompt must systematically include four critical elements: the goal, the context, the constraints, and the definition of done.By structuring prompts in this manner, the operator heavily reduces the assumptions the LLM must make, scoping the generation strictly to the task at hand. For AutoResearch-RL, a tactical prompt during an interactive debugging session might look like this:Goal: Diagnose and resolve the out-of-memory (OOM) error occurring during the backward pass when the MLP expansion ratio is increased to 4x.Context: Review the seed/train_gpt.py script, specifically the TransformerBlock class. Review the recent failure trace in experiment_logs.jsonl.Constraints: You must maintain the 16 MB artifact limit. Do not reduce the vocabulary size to compensate. Implement gradient checkpointing if necessary. Utilize the AST diff parser.Done When: python3 main.py --iterations=1 completes the first 100 steps without throwing an OOM exception, and the artifact size remains compliant.Furthermore, selecting the appropriate reasoning effort level is paramount. For fast, well-scoped syntax fixes, a low or medium reasoning level accelerates output. However, for complex architectural modifications and debugging the multi-step MDP loop, high or xhigh reasoning effort must be specified. The xhigh setting forces the agent to exhaustively map out the implications of a change before generating the code, which is essential when a single structural error could waste ten minutes of valuable H100 GPU compute.9. Local LLM Integration (OSS Mode)While cloud models like gpt-5.3-codex or gpt-5.1-codex-max provide superior reasoning capabilities, reliance on a cloud API introduces network latency, rate limits, and significant operational costs. These issues scale poorly for perpetual research loops generating hundreds of AST diffs and analyzing vast amounts of telemetry data overnight. CI/CD Orchestration and Headless Execution (codex exec)To realize the ultimate vision of AutoResearch-RL—a fully unsupervised, perpetual algorithmic discovery system—the agent loop must be decoupled from the interactive terminal interface and embedded directly into continuous integration (CI) pipelines or orchestration layers.The codex exec subcommand is engineered specifically for scripted, non-interactive environments. It accepts input via standard input (stdin) or command-line arguments and can output structured JSON schemas, making it ideal for interacting with the AST diff parser.11.1 Headless Automation Wrapper ScriptThe following bash script demonstrates a robust wrapper for invoking Codex as a headless evaluation engine within the AutoResearch-RL pipeline. It utilizes codex exec, pipes the output directly into the AST parser, and ensures complete automation by dangerously bypassing approvals within a hardened, isolated Docker container using the --dangerously-bypass-approvals-and-sandbox (or --yolo) flag.Bash#!/bin/bash
# run_autoresearch.sh - Headless RL Loop Execution Wrapper
# Executes the perpetual AutoResearch-RL loop utilizing the Codex CLI in headless mode.

set -e

# Ensure dependencies are pristine and base tests pass
pip install -q -r requirements.txt
pytest -q tests/

ITERATIONS=${1:-300}
CURRENT_ITER=0

while; do
    echo "Starting MDP Iteration: $CURRENT_ITER"
    
    # Generate AST Patch via Codex Headless Exec
    # Utilizing an explicitly defined output schema to ensure strict JSON compliance
    codex exec \
        --profile local-autoresearch \
        --dangerously-bypass-approvals-and-sandbox \
        --output-schema codex-output-schema.json \
        "Review experiment_logs.jsonl. The last reward decreased indicating stagnation. Propose a structural change to train_gpt.py modifying the Multi-Head Attention mechanism to utilize Cross-Sequence Attention. Return the patch in strict AST-diff JSON format." > current_patch.json
    
    # Parse and apply the generated AST Diff
    python3 agent/ppo_agent.py --apply current_patch.json
    
    # Run Static Causality Audit prior to GPU dispatch
    if! python3 auditor/causality_auditor.py seed/train_gpt.py; then
        echo "Causality leak detected by static analysis. Reverting patch to maintain integrity."
        git checkout seed/train_gpt.py
        continue
    fi
    
    # Execute Training Loop with a strict 10-minute timeout mimicking Parameter-Golf rules
    timeout 600s torchrun --standalone --nproc_per_node=8 main.py --iterations=1
    
    # The environment updates policy network weights internally based on the val_bpb outcome.
    # Telemetry is written to experiment_logs.jsonl for the next iteration.
    
    CURRENT_ITER=$((CURRENT_ITER+1))
done
